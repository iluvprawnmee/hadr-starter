§ 01

What this codebase is

Inventory · 14 tracked files

This is not an application. It is a brief — a three-day course template for building an autonomous monitoring agent for humanitarian assistance and disaster response, shipped deliberately empty of code.

The end state is specified precisely: an agent that watches three live disaster feeds, filters noise, assesses what remains, and publishes a morning situation report to dashboard.html at 08:30 Singapore time — unattended, and silent when nothing has changed. The README is explicit that how is specified nowhere.

What the repository does contain is scaffolding for the way of working: a conventions file the agent must obey (CLAUDE.md — currently blank), an agent-kept decision log, a solutions directory so no debugging lesson is paid for twice, and a disabled GitHub Actions workflow whose comments encode the intended architecture before any code exists.

The three feed notes are the real payload. Each carries a verified endpoint, a truncated live response, and three open questions that are, in effect, the hard problems of the system: event identity, upstream revision, and failure behaviour.

USGS

GeoJSON · rolling windows · no auth

Real-time earthquakes, regenerated every minute. The cleanest feed: stable schema, machine timestamps, significance scores.

Events get revised — magnitude, location, occasionally deleted outright — after publication.

GDACS

GeoJSON + RSS · multi-hazard · no auth

EU/UN multi-hazard alerts with colour-coded severity. Broadest coverage: quakes, cyclones, floods, volcanoes, drought, fire.

Four different alert fields per event; no published rate limits or uptime guarantees.

ReliefWeb

API (gated) · RSS (open) · curated

UN OCHA's human-curated disaster registry. Slow-moving and authoritative — an event appears once humans decide it matters.

API requires a pre-approved appname (403 without); approval may not arrive this week.

§ 02

What we would build first

One slice · end to end · runnable today

The thinnest path from a live feed to a committed dashboard.html — run by hand, once.

Fetch

Pull USGS all_day.geojson. One feed only. Follow redirects, log the final URL, save the raw payload.

Filter

A deterministic script in scripts/ keeps events above a significance threshold. No model call. Same input, same answer, twice.

Assess

One headless model pass over the survivors: what happened, where, how bad, who is affected. Structured output, not prose soup.

Publish

Render dashboard.html and commit it. Ugly is fine. Committed and reachable is the definition of done.

Why this slice, why USGS

  1. It proves the whole pipeline shape — fetch, deterministic gate, model assessment, publish — which is exactly the architecture sitrep.yml.disabled already prescribes: the script decides whether anything changed; the model never decides whether to wake up.
  2. USGS is the lowest-risk feed: no auth, no approval queue, refreshed every minute, and honest about its own revisions. GDACS adds hazard breadth later; ReliefWeb waits on an appname.
  3. It produces the product on day one. A dashboard that exists and is wrong is reviewable; a pipeline that is elegant and unpublished is not.
  4. Every hard problem gets a seam — dedup, change detection, and scheduling each slot into an existing stage instead of forcing a rewrite.

Deliberately out of scope

  • Cross-feed deduplication — one feed means the same-event problem cannot exist yet
  • Scheduling & the 08:30 cron — the workflow stays disabled until both TODO steps exist
  • Change detection / staying quiet — meaningless until there are two consecutive runs to compare
  • ReliefWeb appname request — submit the form on day one, build against RSS meanwhile
  • Dashboard design — layout is a Day 3 trust problem, not a Day 2 autonomy problem
¶

First commit before the slice: fill in CLAUDE.md. The template itself warns that an empty conventions file is also a decision — just not one you made. Language, test command, and deviations policy are upstream of every line the agent will write; choosing them after the first slice means rewriting the first slice.

§ 03

Ten questions before we start

Answers wanted in writing · they become goal.md
Purpose 01

Who reads the 08:30 report, and what decision does it drive?

A duty officer deciding whether to wake a response team needs different depth, ordering, and tone than an executive skimming over coffee. Everything downstream — length, layout, what counts as "the headline" — hangs on this.

Scope 02

What is the geographic and hazard scope?

Global everything, or a region of interest? The 08:30 Singapore anchor hints at Asia-Pacific. And are slow-onset hazards — drought, disease — in scope, or only sudden events?

Threshold 03

Where is the line between signal and noise?

GDACS Green events arrive constantly; USGS logs two hundred quakes a day. Is the bar GDACS Orange+? A USGS sig score? Population within range? We need the reportability rule stated, not inferred.

Identity 04

When two feeds report the same disaster, which record wins?

The same physical earthquake arrives from USGS and GDACS under different IDs, and from ReliefWeb days later under a GLIDE number. What makes two records the same event, and whose ID do we store as canonical?

Revision 05

What happens to a published report when its event changes underneath it?

USGS revises magnitudes and occasionally deletes events; GDACS escalates colours after first report. Silently correct, visibly annotate, or leave the published record as history?

Quiet 06

What exactly counts as "nothing has changed"?

New event only? An alert-level escalation on a known event? A casualty figure revised upward? The deterministic gate in scripts/ needs this defined precisely — it is the one function that must never guess.

Failure 07

On a morning a feed is down, does the report still publish?

A missing report and a report that says "GDACS unreachable since 03:00, coverage degraded" are very different products. Which failure is acceptable — silence, or a report with a stated data gap?

Access 08

Has a ReliefWeb appname been requested yet?

Approval takes time we may not have this week. If not — we submit the form today and build against the open RSS feed, accepting that RSS lacks the API's filtering and structured fields. Confirm that trade-off is acceptable.

Runtime 09

Does this run on GitHub Actions, and with what budget?

sitrep.yml.disabled implies Actions plus headless Claude. Confirm: is that the deployment target, whose API key funds the model calls, and are there minutes or cost ceilings that shape how often we may poll?

Conventions 10

What goes in CLAUDE.md — language, test command, deviations policy?

All four sections are blank, and the template warns that an empty conventions file is a decision by default. Python or TypeScript, what a passing test run looks like, and how far the agent may deviate before it must stop and ask.
