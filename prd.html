HADR Monitor

An unattended agent that turns three noisy disaster feeds into one trustworthy morning situation report.

Region · Asia-Pacific Feeds · GDACS · USGS · ReliefWeb Publishes · 08:30 SGT → dashboard.html Stack · Python 3.12 · SQLite · GitHub Actions
01

The goal

Every morning at 08:30 Singapore time, publish one ranked situation report of the Asia-Pacific natural-hazard events that are genuinely new or changed since the last report — filtering out the ~95% of feed volume that is noise, reading as one story per real-world crisis, staying quiet but never silent on a calm morning, flagging itself honestly when a feed is down, issuing corrections when a prior headline is downgraded or deleted, and running unattended.

The hard part is not fetching — it is the diff engine: an editorial policy, backed by a persistent store, that decides what counts as news. The feeds are rolling windows with no history and no deletion tombstones, so the agent must keep the memory they do not.

03

The data model

A per-entity tracker in SQLite (the persistent memory the feeds lack), plus a raw archive written before parsing. Severity is stored on the project's own house scale — NOTABLE SERIOUS CRITICAL — never a feed's native colour.

TableRepresentsKey fields
raw_payload Every fetch, archived before parsing (schema-drift & fixture source) id · feed · fetched_at(UTC) · http_status · body
event One tracked hazard Event per feed record (a lifecycle entity) id · feed · feed_key · hazard_type · occurred_at(UTC) · lat · lon · depth · magnitude · feed_alert · house_severity · country · iso3 · glide · first_seen · last_seen_poll · lifecycle_status · raw_ref
episode Updates to a fast-moving Event (cyclone track, flood re-issue) id · event_id → · episode_key · updated_at(UTC) · alert_level
situation Cross-feed aggregate — the unit the report tells as one story id · glide · hazard_type · house_severity(max) · centroid_lat · centroid_lon · window_start
situation_member Many-to-one link: several Events → one Situation situation_id → · event_id → · role
report Each published morning report id · published_at(UTC) · window_start · window_end
news_item What appeared in a report, and why id · report_id → · situation_id → · news_type · house_severity · rank
feed_health Per-feed freshness driving the stale-data banner feed · last_success_at(UTC) · last_status

Relationships  ·  event *→1 situation (via situation_member)  ·  situation 1→* news_item across reports  ·  event 1→* episode  ·  every event & parse traces back to a raw_payload.

news_type ∈ { new · escalation · revision · episode · correction }  ·  lifecycle_status ∈ { active · downgraded · deleted }

04

Out of scope

  • All-hazard coverage — conflict, epidemics, industrial accidents, heatwaves, landslides. The three feeds cannot deliver them.
  • Real tsunami warnings — tsunami:1 is only a NOAA-zone flag; PTWC / tsunami.gov are separate sources. Not integrated.
  • Global coverage outside the Asia-Pacific bounding box.
  • Sub-floor reporting — GDACS Green, PAGER green/none, tiny quakes. Sole exception: an in-region M≥6.5, depth≤70km quake with no PAGER yet.
  • ReliefWeb casualty & sitrep detail — blocked on the appname-gated reports API. MVP surfaces disaster declarations / GLIDE only.
  • Historical backfill beyond what the agent's own store accumulates from first run.
